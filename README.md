# Шаг 1. Входная фиксация сегмента и сценария

## Сегмент
**Руководители писательских коллективов и малые студии (2–10 человек)**, работающие над интерактивной литературой (книги-игры), сценариями для игр или образовательными кейсами с ветвлением сюжета.

## Сценарий использования
Команда авторов работает над нелинейным сюжетом. Один автор пишет главу, другой создаёт альтернативную ветку, третий редактирует. Им нужно видеть общую структуру, избегать конфликтов версий и согласовывать продолжения.

## Основная проблема
- нет инструментов для **командной работы** над ветвящимися сюжетами
- связи между главами приходится отслеживать **вручную**
- голосование за лучшие продолжения идёт в Telegram / мессенджерах — теряется контекст
- структура истории (кто, за кем, куда ведёт) быстро становится **хаотичной**

## Последствия
- трата времени на ручное согласование версий (до 30% времени)
- появление "битых" ссылок и потерянных глав
- сюжетные противоречия и дублирование веток
- снижение продуктивности команды

## Текущая замещающая альтернатива

Сейчас команды решают проблему через:

- **Google Docs / Notion** — конфликты версий, нет визуализации структуры
- **Miro / Draw.io** — ручное рисование графа, которое устаревает через день
- **Telegram / Discord** — голосование в отрыве от контекста
- **Twine** — нет командной работы
- **Excel / таблицы** — для отслеживания связей (крайне неудобно)

## Окна возможностей

Где конкуренты слабы:

- **Нет инструментов для командной работы** над ветвящимися сюжетами
- **Нет автоматической визуализации** структуры в реальном времени
- **Дорогие решения** (Articy) недоступны малым командам
- **Бесплатные решения** (Twine) не поддерживают совместную работу

# Шаг 3. Критерии выбора сегмента

Основные критерии выбора решения для целевого сегмента:

1. **Возможность командной работы** (одновременное редактирование) ⭐
2. **Автоматическая визуализация структуры** (граф связей) ⭐
3. **Контроль версий и история изменений** ⭐
4. Простота использования (низкий порог входа)
5. Цена / доступность

**Топ-3 критериев:**
- командная работа
- автоматическая визуализация структуры
- контроль версий

# Шаг 4. Гипотеза ценностного предложения

## Вариант 1

Для **руководителей писательских коллективов и малых студий**, которые создают сложные нелинейные сюжеты,  
мы **автоматизируем связывание глав и управление версиями**,  
улучшая **скорость согласования и целостность структуры** по сравнению с Google Docs и Miro,  
за счёт **живой визуализации графа и встроенного голосования**.

## Вариант 2

Для **писательских команд**, которые тратят часы на ручное согласование веток,  
мы **объединяем редактор, визуализацию и голосование в одном окне**,  
сокращая **время на администрирование проекта до 30%** по сравнению с Telegram + таблицы,  
за счёт **автоматического построения связей между главами**.

## Вариант 3

Для **инди-студий и сценарных групп**, работающих над интерактивными историями,  
мы **создаём единую среду для коллективного творчества**,  
улучшая **прозрачность структуры и скорость принятия решений** по сравнению с Twine + мессенджеры,  
за счёт **облачной архитектуры и системы голосования**.

## Основная гипотеза

Для **руководителей писательских коллективов и малых студий**, работающих над ветвящимися сюжетами,  
мы **обеспечиваем автоматическую визуализацию структуры и встроенное голосование**,  
улучшая **скорость согласования и целостность проекта** по сравнению с Google Docs и Miro,  
за счёт **автоматического связывания глав и контроля версий**.

### Отличие от конкурентов

- **Командная работа** из коробки (в отличие от Twine)
- **Автоматическая визуализация** графа в реальном времени (в отличие от Google Docs)
- **Доступная цена** (в отличие от Articy:Draft)

### Почему сегмент должен поверить

- простой и понятный интерфейс
- визуализация структуры сразу после создания глав
- возможность попробовать бесплатно (Freemium)
- решение реальной боли — ручного согласования

# Шаг 5. Определение ключевого функционала (ядро MVP)

## Core (ядро MVP) — то, без чего ценность невозможна

1. **Регистрация и создание команды** (приглашение участников)
2. **Создание глав и веток** (текстовый редактор)
3. **Автоматическая визуализация графа связей** (D3.js)
4. **Контроль версий и история изменений**

## Support — повышает удобство доставки ценности

- профили пользователей и роли (автор, редактор, читатель)
- комментарии к главам
- уведомления о новых ветках и голосованиях
- поиск по главам и веткам
- настройки приватности (публичные / приватные истории)

## Later — можно отложить

- интеграция с игровыми движками (Unity, Unreal)
- система рекомендаций и подборок
- мобильное приложение
- API для сторонних разработчиков

# Шаг 6. Гипотезы проверки ценности

## Гипотеза 1
Руководители писательских коллективов испытывают проблему ручного согласования веток и конфликтов версий.

**Метод проверки:**  
глубинные интервью с 10–15 руководителями писательских групп, сценаристами, нарративными дизайнерами.

**Критерий подтверждения:**  
≥70% респондентов подтверждают проблему и описывают её как "болезненную".


## Гипотеза 2
Автоматическая визуализация структуры (граф связей) является ключевой ценностью для команд.

**Метод проверки:**  
демонстрация прототипа / кликабельного макета целевой аудитории.

**Критерий подтверждения:**  
≥60% респондентов называют визуализацию "очень полезной" или "необходимой".


## Гипотеза 3
Команды готовы платить за решение, которое экономит время на согласование.

**Метод проверки:**  
лендинг с описанием тарифов (Free, Premium $9, Team $29) и замер конверсии в регистрацию / клик на платный тариф.

**Критерий подтверждения:**  
конверсия в целевое действие ≥5% (или готовность назвать цену в интервью).


## Гипотеза 4
Встроенное голосование за продолжение сюжета ускоряет принятие решений в команде.

**Метод проверки:**  
опрос пользователей после тестирования прототипа / MVP.

**Критерий подтверждения:**  
≥50% респондентов называют функцию голосования "значительно ускоряющей работу".


## Гипотеза 5
Писательские коллективы готовы перейти с бесплатных инструментов (Google Docs, Twine) на платформу ради командных функций.

**Метод проверки:**  
сравнительный анализ + опрос о готовности сменить инструмент.

**Критерий подтверждения:**  
≥40% респондентов готовы рассмотреть переход при наличии бесплатного тарифа.
